DIAGNOSIS THE STARTING POINT
xgeeks 002 / 010

Where this proposal lands

Intellectually serious.
Architecturally conservative.

The current proposal is a governance instinct expressed in agentic language. It hardens the floor of AI usage at Wellhub. It does not yet raise the ceiling.

What it is

A Go CLI cia wrapping structured prompts as Markdown templates, bound to each SDLC stage. Versioned, reviewed, gated.

What it isn't

An agentic system. There are no agents in the design. No autonomous loops, no tool use, no self-correction, no initiation by AI.

Why it matters

It is the right first chapter. It is not yet the book. AI-native organisations differ by what AI does, not what humans add around it.

001 STRENGTHS TO PRESERVE
xgeeks 003 / 010

What's working

Six decisions that
should survive any redesign.

The proposal shows unusual discipline for an AI initiative. These are the ideas that are already aligned with how AI-native organisations operate. Keep them.

01
Prompts as first-class artifacts
Versioned, reviewed like code, kept in two-way sync with the codebase. The single strongest idea in the document.
02
Bound to the existing SDLC
Commands map to real stages: RFC, DB migrations, deploy windows, post-deploy, on-call. Not a parallel sandbox.
03
A disciplined Non-Goals section
Hotfixes, spikes, and aesthetic work explicitly out of scope. No bypassing CODEOWNERS, Infosec, Privacy, Fraud.
04
The Code Red audit story
Every AI-assisted change traceable back to the prompt that produced it. Exactly what regulators and auditors want.
05
Norms & Safeguards catalog
Team conventions and non-negotiables in a governed, reusable asset. Behaves like organisational "house rules" for any AI workflow.
06
Operationally sane choices
Single Go binary, model-agnostic, gateway-mediated, per-service config, central templates pinned by version. Shippable.

002 CRITICAL GAPS
xgeeks 004 / 010

Where the design falls short

Ten construction notes,
ordered by impact.

2.1

No agents in the design

Titled "SDLC - Agents," but architecturally a CLI that fills templates for human review. No plan-act-observe loop, no tool use, no initiation by AI.

2.2

AI as generator, not collaborator

A CLI invoked deliberately before doing work is the wrong surface area for the gains that actually happen in the editor, in the PR, and in operations.

2.3

Solves the wrong bottleneck

The Abstract names ambiguity, review load, inconsistency, and prod risk. The remedy adds artifacts humans must produce and review. Shifts the load. Does not reduce it.

2.4

No evaluation, no learning loop

Versioning a prompt does not mean improving it. The "compounding asset" claim has no measurement behind it. The single highest-leverage capability missing.

2.5

No tool layer (MCP or equivalent)

Integrations are scripts wired into one binary. The industry is converging on MCP. Capabilities are trapped inside the CLI.

2.6

No autonomy gradient

Every command sits at copilot. No articulated path to autopilot or autonomous, which is where most enterprise AI programmes stall.

2.7

Security model needs depth

"The gateway redacts what needs redacting" is a sentence, not a design. No prompt-injection story, no credentials story.

2.8

Excluded work is most of the work

Hotfixes, spikes, exploratory work out of scope. In any real organisation those modes are a large share of throughput. Engineers will route around SPDD for all of it.

2.9

Several scripts should be agents

cia post-deploy, cia alert-coverage, cia deploy-window, cia db-query-plan: bounded, low-blast-radius loops that should run continuously, not be invoked.

2.10

No adoption or outcome plan

No pilot scope, no rollout sequence, no token-spend projection, no commitment to which numbers will move. The most addressable gap.

003 POSITIONING
xgeeks 005 / 010

Three postures, routinely conflated

AI-assisted is not
AI-augmented is not
AI-native.

Here
AI-assisted

Humans author everything. AI is invoked in moments, on demand.

Drafts, suggests, autocompletes, always inside a human-driven loop.

Adjacent
AI-augmented

Humans set intent, review, approve. AI runs in the editor and in CI continuously.

Generates whole diffs, runs reviews, fixes lints, drafts RFCs, files alerts.

Far
AI-native

Humans set direction, define guardrails, approve high-stakes actions, handle escalations.

AI operates as a teammate: opens PRs, triages alerts, summarises incidents, owns recurring loops within bounded permissions.

The proposal's mental model: the engineer is the author, the AI is a faster typewriter governed by templates. An AI-native model inverts that default: the agent is the author of routine artifacts, the engineer is the editor in chief.

005 THE AI-NATIVE VERSION
xgeeks 007 / 010

A picture of the inverted default

Same SDLC stages.
Different authors.

Agents that do real work
01
PR-review agent
Runs on every PR before a human looks. Applies the DB-migration, public-endpoint, and observability-coverage checklists, plus Norms.
02
Alert-coverage agent
Lives continuously against the canonical monitor catalog, opens PRs for gaps, owns the conversation, closes itself out.
03
Post-deploy agent
Watches Argo, Datadog, Grafana for the one-hour window automatically, summarises Golden Signals, escalates only when needed.
04
On-call triage agent
Reads PagerDuty, Datadog, Loki at incident start, drafts the timeline, proposes the rollback, links the runbook.
05
Dependency agent
Classifies Dependabot risk, opens curated PRs, runs the test plan, escalates only the ambiguous ones.
06
IDE agent
Knows the REASONS Canvas, Norms, and Safeguards by default. Artifact discipline shows up where engineers already work.
Platform beneath the agents
MCP tool layer
Internal systems (Datadog, Argo, Vault, IAM, Sonarcloud, the gympass-app catalog, Production Queries sheet) exposed as typed tools that any agent can call.
Evaluation flywheel
Golden datasets per agent, regression CI on prompt and tool changes, quality dashboards visible to leadership, feedback ingested from real PRs and incidents.
Agent governance model
Codifies the autonomy gradient (copilot, autopilot, autonomous). Permissions, blast-radius limits, audit logging, rollback. The artefact Infosec, Privacy, and Fraud will need.

006 THE PATH FORWARD
xgeeks 008 / 010

Five high-leverage moves

Sequenced.
Each independently valuable.

Each move is tied to a measurable outcome. Each one is defensible at leadership level. None of them requires discarding the current proposal.

01
Reframe around outcomes, not artifacts
PR cycle time, change failure rate, on-call hours per incident. Make the programme falsifiable in numbers leadership can evaluate.
02
Move to in-IDE and in-PR agents
Keep artifact discipline as shared context. Add an agentic PR reviewer that runs before humans. Governance comes to engineers, not the other way around.
03
Build the evaluation flywheel first
Golden datasets, regression CI, quality dashboards, feedback ingestion from PRs and incidents. Without this, no agent earns more autonomy than day one.
04
Pilot one autonomous agent
Alert-coverage gap-filler, dependency triage, or on-call summariser. High volume, low blast radius. One agent in production is worth ten decks.
05
MCP layer + governance, in parallel
Typed tools any agent can call. Autonomy gradient codified per agent. Permissions, audit, rollback. The substrate that turns one pilot into a fleet.
